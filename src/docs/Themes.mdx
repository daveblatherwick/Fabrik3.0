# Themes

Fabric combines three independent mode axes. Each axis is controlled by a `data-*` attribute on the document root and cascades to every component via CSS custom properties — no JS runtime theming.

| Axis    | Values                        | Attribute      |
| ------- | ----------------------------- | -------------- |
| Brand   | Adaptive / Aeron / Demo       | `data-brand`   |
| Color   | Light / Dark                  | `data-theme`   |
| Density | Compact / Default / Spacious  | `data-density` |

That's 3 × 2 × 3 = **18 theme combinations** from a single component implementation. Use the toolbar above to try them.

## How it works

At `:root` the default tokens apply. Each `[data-*]` selector overrides only the subset it owns:

```css
:root, [data-brand="adaptive"] { --fabric-brand-600: #676AC6; ... }
[data-brand="aeron"]            { --fabric-brand-600: #1BA768; ... }

:root, [data-theme="light"]    { --fabric-bg-primary: #FFFFFF; --fabric-text-primary: #141414; ... }
[data-theme="dark"]             { --fabric-bg-primary: #0A0A0A; --fabric-text-primary: #F7F7F7; ... }

:root, [data-density="default"] { --fabric-spacing-md: 8px; --fabric-text-sm: 11px; ... }
[data-density="compact"]        { --fabric-spacing-md: 6px; --fabric-text-sm: 10px; ... }
```

Setting attributes on `<html>` switches everything at once. The components themselves never mention brands, modes, or densities — they only reference semantic tokens.

## When this model breaks

- **Custom fintech components** (trading tables, currency tiles, order tickets) often need brand-specific data density decisions that don't fit the `data-density` axis — expect per-brand overrides.
- **Per-component theme overrides** (one button on one screen is a different color) need explicit escape hatches, not theme changes.
- **Motion and behavior** aren't currently tokenised here — if the design system needs to govern transitions, that's a new axis.
